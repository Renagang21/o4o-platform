## 1. 문서 목적

본 문서는 o4o 플랫폼에서 정의한 **Content Core**가
각 서비스(CMS / LMS / Signage / Extension)에서
**어떻게 "소비(consumption)"되어야 하는지**를 명확히 규정한다.

이 문서는 다음을 목적으로 한다.

* Content Core의 **역할 오해 방지**
* 서비스별 임의 해석·구현 차단
* 파트너 협업 이전 **플랫폼 기준 고정**
* "무엇을 하지 말아야 하는지"를 명확히 선언

---

## 2. Content Core의 역할 재선언 (중요)

### 2.1 Content Core가 **아닌 것**

Content Core는 다음을 **절대 수행하지 않는다**.

* 콘텐츠 생성 ❌
* 콘텐츠 수정/삭제 ❌
* 승인/차단/판단 ❌
* DB Source of Truth ❌
* 서비스 로직 포함 ❌

---

### 2.2 Content Core가 **하는 것**

Content Core는 오직 다음 역할만 수행한다.

* 콘텐츠를 **의미 단위로 표현**한다
* 콘텐츠를 **공통 enum / 정책 기준으로 분류**한다
* 콘텐츠를 **읽기(Read-only) 기준 모델**로 제공한다
* 여러 서비스가 **같은 언어로 콘텐츠를 이해하도록 돕는다**

👉 Content Core는
**"콘텐츠 저장소"가 아니라
"콘텐츠 공통 해석 체계"**이다.

---

## 3. 소비(Consumption)의 정의

본 문서에서 말하는 "소비"란 다음을 의미한다.

> **기존 서비스가 보유한 데이터를
> Content Core 기준으로 '읽어 해석하고 활용하는 행위'**

### 소비에 포함되는 행위

* 목록 조회
* 상세 조회
* 선택
* 참조
* 배치(미래)

### 소비에 포함되지 않는 행위

* 생성
* 수정
* 삭제
* 상태 변경
* 정책 재정의

---

## 4. 공통 소비 원칙 (모든 서비스에 적용)

모든 서비스는 Content Core를 소비할 때
**아래 원칙을 반드시 지켜야 한다.**

### 원칙 1. Read-only 원칙

* 모든 소비는 읽기 전용이다
* Content Core를 통해 **어떠한 쓰기 동작도 수행할 수 없다**

### 원칙 2. Source of Truth 비침해

* 원본 데이터의 진실은 **각 서비스 DB에 있다**
* Content Core는 원본을 대체하지 않는다

### 원칙 3. 의미적 매핑만 허용

* 필드 복제 ❌
* 데이터 동기화 ❌
* 파생 테이블 ❌
* **의미(enum/상태/소유권)만 참조**

### 원칙 4. 정책 우선

* Visibility / Ownership / Status 해석은
  **Content Core 정책 기준을 따른다**
* 서비스별 임의 해석 금지

## 5. 서비스별 소비 시나리오 (개념 수준)

> ⚠️ 이 장은 **구현 지시가 아니다**
> "가능한 소비 방식"을 개념적으로 정의한다.

---

### 5.1 CMS의 소비 시나리오

**역할**

* 콘텐츠의 실제 저장 주체
* Media/File의 Source of Truth

**소비 방식**

* CMS Media를 Content Core 기준으로 해석
* ContentType / Status / Visibility 기준 표시
* 운영자에게 "이 미디어는 어떤 콘텐츠인가"를 설명

**금지**

* Content Core 기준으로 CMS 데이터를 변경 ❌
* Content Core 상태를 CMS 상태로 강제 ❌

---

### 5.2 LMS의 소비 시나리오

**역할**

* 교육/학습 맥락에서 콘텐츠 활용

**소비 방식**

* 콘텐츠를 "학습 자료로 사용 가능 여부" 판단
* Visibility / Ownership 기준으로 노출 제어
* 콘텐츠 자체는 수정하지 않음

**금지**

* LMS가 콘텐츠를 소유했다고 해석 ❌
* Content Core를 학습 전용 모델로 확장 ❌

---

### 5.3 Signage의 소비 시나리오 (미래형)

**역할**

* 디스플레이 채널에서 콘텐츠 활용

**가능한 소비**

* Content Asset을 재생 가능한 리소스로 참조
* Visibility / Status 기준으로 사용 가능 여부 판단
* Playlist/Channel에 **참조(reference)** 형태로 연결

**아직 하지 않는 것**

* 실제 선택 UI ❌
* 자동 배치 ❌
* 파트너 노출 ❌

👉 이 시나리오는
**사업자·파트너 협업 단계에서만 구현 가능**하다.

---

### 5.4 Extension의 소비 시나리오

**역할**

* 산업별 특화 로직 제공

**소비 방식**

* Content Core enum을 기준 언어로 사용
* 콘텐츠를 직접 소유하지 않음
* 항상 Core 기준에 종속

**금지**

* Extension이 자체 Content 기준 정의 ❌
* Core enum 재정의 ❌

---

## 6. 명시적 금지 사항 (Hard Ban)

다음은 **버전 업 없이는 절대 허용되지 않는다.**

* Content Core를 CRUD 대상으로 사용하는 행위
* Content Core에 전용 DB 테이블 추가
* 서비스가 Content Core를 "주 시스템"으로 오해
* 소비 과정에서 원본 데이터 변경
* "편의를 위한" 임시 구조 도입
