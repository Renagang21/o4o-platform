## 0. 한 줄 요약

초기 유통참여형 펀딩은 **시스템 자동화를 최소화**하고, **제품 개발자(공급자)가 포럼을 운영**하며 **Neture 운영자가 송금(돈의 흐름)을 책임지는** 사람 중심 운영 모델로 운영한다. 참여자 제한, 포럼 자동 gate, PG/checkout, 상태 자동화는 **실제 운영 필요성이 확인될 때까지 보류**한다.

---

## 1. 운영 원칙 (확정 — 2026-06-06)

```
초기 유통참여형 펀딩은 참여 신청자를 시스템이 강하게 제한하지 않는다.
제품 개발자가 신청자를 검토하고, 포럼 승인도 제품 개발자가 운영한다.
참여자는 포럼에서 제품 정보와 조건을 확인한 뒤 송금 여부를 결정할 수 있다.
송금은 Neture 운영자가 수령하며, 송금 완료자 명단을 제품 개발자에게 공유한다.
일정 기간 내 송금하지 않은 참여자는 제품 개발자가 포럼에서 탈퇴시킬 수 있다.
PG/checkout 및 포럼 자동 gate는 실제 운영 필요성이 확인될 때까지 보류한다.
```

---

## 2. 초기 흐름 (사람 중심, 자동화 최소)

```text
1. Neture 사용자가 유통참여형 펀딩에 관심/참여 신청
2. 제품 개발자가 신청자를 확인
3. 제품 개발자가 적합하다고 보는 사람을 포럼에 승인
4. 포럼에서 제품 설명, 개발 진행, 조건, 리워드, 정산 방식 안내
5. 참여자는 포럼 내용을 보고 송금 여부를 결정
6. 송금은 Neture 운영자가 받음
7. Neture 운영자가 송금 완료자 명단을 제품 개발자에게 전달
8. 제품 개발자는 일정 기간 내 미송금자를 포럼에서 제외
9. 송금 완료자 중심으로 개발 진행, 피드백, 제품 정산 진행
```

---

## 3. 역할 분담

| 구분 | 초기 운영 방식 | 주체 |
|---|---|---|
| 참여 신청 | Neture 사용자 누구나 신청 가능 (시스템 제한 없음) | 참여자 |
| 제품 적합성 판단 | 제품 개발자가 판단 | 제품 개발자(공급자) |
| 포럼 승인 | 제품 개발자가 포럼 운영자로서 승인 | 제품 개발자 |
| 포럼 이용 | 송금 전에도 일정 기간 이용 가능 | 참여자 |
| 송금 결정 | 포럼에서 제품 개발 상황·조건을 보고 참여자가 결정 | 참여자 |
| 송금 수령 | Neture 운영자가 수령 (돈의 흐름 통제) | Neture 운영자 |
| 송금 내역 공유 | Neture 운영자가 제품 개발자에게 전달 | Neture 운영자 |
| 미송금자 처리 | 제품 개발자가 일정 기간 후 포럼에서 탈퇴 처리 | 제품 개발자 |
| 시스템 자동화 | 초기에는 최소화 | — |

---

## 4. 이 모델의 근거

1. **초기 운영이 단순하다.** 신청/승인/송금 확인/탈퇴를 각각 사람이 판단 → 복잡한 상태머신 불필요.
2. **제품 개발자가 커뮤니티 운영 주체가 된다.** 제품 개발자와 초기 참여자가 함께 제품을 이해·검토하는 구조이므로 포럼 운영은 제품 개발자가 맡는 것이 자연스럽다.
3. **참여자는 송금 전에 충분히 판단할 수 있다.** 포럼에서 개발 방향·정산 조건·일정·제품 구성·공급 조건을 본 뒤 송금 → 참여자 부담 감소.
4. **Neture가 돈의 흐름을 통제한다.** 송금을 Neture 운영자가 받아 제품 개발자의 사기성 운영 위험을 줄인다(초기 신뢰 확보 핵심).
5. **나중에 자동화하기 쉽다.** 수동 운영으로 실제 필요한 상태값·화면을 확인한 뒤 PG/포럼 자동 gate를 붙인다.

---

## 5. 보류 / 제외 (초기 단계에서 개발하지 않음)

| 항목 | 처리 | 사유 |
|---|---|---|
| 참여 대상 제한(자격 gate, IR G3) | **보류** | 초기엔 누구나 신청 → 제품 개발자 검토 + 포럼 승인으로 사람-게이트. 시스템 제한은 운영 유연성을 떨어뜨림 |
| 포럼 접근 자동 gate | **보류** | 포럼 승인/탈퇴는 제품 개발자 수동 운영 |
| 제품 개발자 승인 + Neture 입금확인 자동 상태머신 | **보류** | 송금 확인·명단 공유는 운영자 수기 |
| PG / checkout (IR G4) | **장기 보류** | 송금은 Neture 운영자 오프라인 수령. 실제 필요성 확인 후 별도 고도화 |
| 상태 자동화 일반 | **최소화** | 수동 운영으로 필요 상태값 검증 후 단계적 자동화 |

> ⚠️ 위 항목들은 "미구현 결함"이 아니라 **의도된 초기 운영 모델**이다. IR 의 "모집·참여·승인 + 운영자 수기 정산·이행 스켈레톤" 판정은 본 모델 관점에서 **초기 운영 모델에 부합**으로 재해석한다. (단, 보안 경계는 예외 — 무인증 노출은 이미 OFFLINE-OPERATION-SAFETY-V1 으로 해결)

---

## 6. 지금 필요한 것 (후속 우선순위)

1. **완료**: 보안 핫픽스 + 오프라인 운영 고지 (`OFFLINE-OPERATION-SAFETY-V1`)
2. **보류**: 참여 대상 제한(G3) / 포럼 자동 gate / 입금확인 자동 상태머신
3. **다음**: **공급자 설계 필드 보강(G5)** — 제품 개발자가 목표 매장 수, 1인당 참여 권장 금액, 제품 정산 기준, 정산 제품 구성, **송금 후 포럼 운영 방식**, **미송금자 처리 기준**을 명확히 작성할 수 있게. **1차는 설명 필드·안내 중심**(DB 대규모 변경 지양)
4. **그다음**: **운영자 송금 내역 관리** — Neture 운영자의 송금 수령·완료자 명단 관리·제품 개발자 공유
5. **그다음**: 매장 랜딩 추적(G6)
6. **장기**: PG/checkout(G4), 포럼 자동 연동

---

## 7. 불변 / 보안 경계

- **돈의 흐름은 Neture 운영자가 통제** — 제품 개발자가 직접 참여금을 수령하지 않는다.
- **보안 경계는 운영 모델과 무관하게 항상 유지** — `participationId` 단독으로 배송지(PII)/이행 정보에 접근 불가(인증 + 참여 당사자/운영자 권한). PG 미사용 초기에도 필수. (`OFFLINE-OPERATION-SAFETY-V1` 에서 적용·검증 완료)
- 내부 Market Trial 코드명/라우트/API/DB 식별자 불변. 외부 표기 유통참여형 펀딩 유지.
