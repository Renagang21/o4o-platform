## 1. Executive Summary

### 결론 한 줄
**처방·일반의약품은 정부 파일(CSV/Excel) 기반 전체 동기화(FULL_SYNC)가 명확히 가능하고 우월하다. 비의약품은 상품군별로 갈린다 — 의료기기만 바코드 자동등록이 되고, 나머지(의약외품·건기식·화장품·식품·위생용품)는 바코드 검색이 불가하여 제품명 기반 온디맨드(ON_DEMAND) 또는 수기(MANUAL)에 머문다.**

### 핵심 발견 (모두 [검증])
1. **의약품 마스터의 백본은 HIRA 약가마스터 파일(data.go.kr `15067462`)이다.** 단일 CSV, **298,183행, 22~23개 컬럼**. `품목기준코드(식약처) · 제품코드(심평원) · 표준코드(13자리) · 일반명코드(주성분코드) · ATC코드 · 전문일반구분 · 취소일자`를 **한 행에 동시 보유**. 5개 요구 연결키를 모두 충족하는 단일 파일.
2. **표준코드(KD-code, 13자리)는 GTIN-13(EAN-13) 구조 그 자체이며 포장단위(per-package)로 부여된다.** 즉 의약품 표준코드 = 그 의약품의 retail 바코드. 1 품목기준코드 → N 표준코드(함량·포장별).
3. **약가/급여 현황의 권위 원천은 HIRA 약제급여목록표(Excel, 월 1회)이다.** 약가마스터는 급여삭제 품목도 누적 유지하므로 **현재 급여 여부 판정 불가** → 월간 목록표로 판정.
4. **식약처↔심평원 공식 매핑은 식약처 묶음의약품정보서비스 API(`15063908`)가 제공**한다(대표 품목기준코드 ↔ 심평원 제품코드 ↔ 주성분코드 ↔ ATC). 약가마스터와 교차검증 가능.
5. **무료 정부 데이터에 "임의 소비재 바코드 → 제품명" 조회는 사실상 존재하지 않는다.** 유일했던 식약처 바코드연계제품정보(`15060549`)·유통바코드(`15064775`)는 **출처가 GS1 Korea(대한상의)이고 2018년 이후 갱신 중단(동결)**. 현행·전수 GTIN 조회는 코리안넷 유료뿐(스코프 외).
6. **예외 — 의료기기.** UDI-DI = GTIN이므로, 의료기기 표준코드 OpenAPI(`15073875`/`15073874`)는 **바코드(UDIDI_CD)로 직접 조회 가능**. 비의약품 중 유일하게 바코드 자동등록에 적합.
7. **전 서비스 무료 + "이용허락범위 제한 없음"**(개발계정 10,000콜/일). 처방의약품 전체저장은 API가 아니라 **파일 다운로드(CSV 연 1회 + Excel 월 1회)**가 정답이며 일일 호출한도 이슈 없음.

### 권장 방식 분포
- **FULL_SYNC**: 처방의약품, 일반의약품
- **ON_DEMAND**: 의약외품, 건강기능식품, 화장품(기능성), 의료기기(바코드 기반)
- **MANUAL_ONLY**: 일반식품·수입식품·위생용품(제품명 보조검색만), 생활용품·안전확인 제품
- **NOT_USEFUL (무료 바코드 조회)**: 일반 소비재 임의 바코드 → 코리안넷 유료 외 경로 없음

## 4. API 접근 및 승인 방법

## 6. 식별자와 연결 키

> 핵심 축: **품목(Product) 단위 식별자** vs **유통/포장(Package) 단위 식별자**. 후자만 retail 바코드와 동일하거나 바코드 그 자체.

| 식별자 | 식별 대상 | 발급 | 형식 | 단위 | GTIN(바코드) 관계 |
|---|---|---|---|:--:|---|
| **식약처 품목기준코드** | 허가받은 의약품 "품목" | MFDS | 숫자(item_seq) | **Product** | 직접 바코드 아님. 1품목→N포장 |
| **의약품 표준코드(KD-code)** | 의약품 판매/포장단위 | MFDS 부여(업체코드=대한상의) | **13자리 = 880+업체4+품목5+검증1 (=EAN-13/GTIN-13)** | **Package** | **= retail 바코드 그 자체**. 전문약은 추가로 GS1-128 |
| **심평원 제품코드(EDI)** | 급여/약가 청구용 제품 | HIRA | 9자리 | Product~Package | 표준코드와 별개. 묶음서비스(A6)/약가마스터(B1)로 매핑 |
| **주성분코드(일반명코드)** | 유효성분(성분+함량+제형) | HIRA | 영숫자 | **성분(다제품 공유)** | 바코드 무관. 동일성분 묶는 키 |
| **GTIN / EAN-13** | 소매 판매단위(SKU) | GS1(국내 GS1 Korea=대한상의) | 13자리 | **Package** | retail 바코드 본체 |
| **의료기기 UDI-DI** | 의료기기 모델(DI 고정) | MFDS(emedi) | GS1/HIBCC 표준(보통 GTIN형) | **Product(모델)** + PI는 포장/생산 | **UDI = DI(GTIN형) + PI**. DI=GTIN |
| **건강기능식품 품목제조관리번호** | 건기식 품목제조신고 1건 | MFDS | 일련번호 | **Product** | 바코드 아님 |
| **식품 품목보고번호** | 식품 품목제조보고 1건 | MFDS/지자체 | 일련번호 | **Product** | 바코드 아님(2018동결 API가 BAR_CD↔보고번호 연결했었음) |
| **의약외품 허가·신고번호** | 의약외품 품목 | MFDS | 허가/신고 일련 | **Product** | 바코드 아님 |
| **화장품 심사·보고번호** | 기능성화장품 품목 | MFDS | 심사=품목기준코드 통지 / 보고=보고일련번호 | **Product** | 바코드 아님. 일반화장품은 번호 자체 없음 |

### 연결키 요약 [검증]
- **의약품 마스터 연결의 중심축 = `품목기준코드(식약처) ↔ 제품코드(심평원/EDI)`.** 주성분코드·ATC가 보조.
- **공식 매핑 제공 2경로**: ① **약가마스터(B1)** — 5개 키가 한 행에 공존, ② **묶음의약품서비스(A6)** — 식약처가 심평원 제품/주성분/ATC를 직접 매핑. → 교차검증 권장.
- **표준코드는 품목기준코드의 per-package 확장**(품목코드 5자리 = 함량 + 포장단위 자리; 끝자리 0=대표코드, 1~9=포장 변형). 따라서 **표준코드 = 의약품 바코드**.
- **의료기기**: UDI/EDI 조회(A15)가 UDI-DI ↔ 품목허가번호 ↔ EDI(요양급여 치료재료코드) 연결.

## 7. 처방의약품 전체 저장 가능성

**판정: FULL_SYNC 가능 · 권장. API 페이지순회보다 정기 FILE 다운로드가 명백히 우월.**

### 7-1. 권위 출처 — 식약처 vs 심평원: **둘 다 필요** [검증/추론]
| | 식약처 허가정보(A1) | 심평원 약제급여목록(B2) |
|---|---|---|
| 모집단 | 허가 전체(전문+일반, 급여+비급여) | **급여 등재 품목만** |
| 비급여 전문약 | 포함 | **미포함** |
| 약가(상한금액) | 없음 | **있음** |
| 표준코드/포장 | 제한적 | 약가마스터(B1)에 누적 |

**권장 구성**: 모집단 row = **식약처 허가정보(A1)** 기준 + 약가/급여상태 attribute = **약제급여목록표(B2)** + 코드연계/표준코드 = **약가마스터(B1) + 묶음의약품(A6)**. 한쪽만으로는 처방의약품 전체저장 불가.

### 7-2. 식약처↔심평원 연결 — 가능, 다중 필드 [검증]
연결키 `품목기준코드 ↔ 제품코드(EDI)`. 약가마스터(B1) 단일 행에 두 코드 + 표준코드 + 주성분코드 + ATC 공존 → 즉시 매핑. 묶음서비스(A6)로 교차검증.

### 7-3. per-package 표준코드/바코드 확보 [검증]
약가마스터(B1)가 **포장단위별 표준코드(13자리 GTIN형)의 1차 출처**. 식약처 유통바코드 API는 stale → 비권장.

### 7-4. 허가취소 / 급여삭제 / 제품명 변경 처리 [검증]
- **허가취소/취하**: 약가마스터 `취소일자`, 묶음서비스 `취소·취하명`. 소프트 삭제(행 보존 + 상태 플래그).
- **급여삭제**: 약가마스터는 삭제 미반영(누적) → **급여 현황은 반드시 월간 약제급여목록표(B2)로 판정**. 직전월 대비 사라진 제품코드 = 급여중지.
- **제품명/양도양수 변경**: `양도양수적용/종료일자` + 한글상품명 변경 → **표준코드/품목기준코드를 안정 PK로**, 제품명은 가변 속성으로 이력 관리.

### 7-5. 초기 적재 + 갱신 전략 [검증]
- **초기 full load**: 약가마스터 CSV 1파일(298,183행) + 최신 약제급여목록표 Excel + 식약처 허가정보 full pull.
- **델타(전용 변경분 API 없음)**:
  - 약가/급여 → **월간 약제급여목록표 Excel 재수신**, 적용일자 기준 diff(신규/상한금액변경/삭제). **FILE이 API보다 우월**.
  - 표준코드 매핑 → 약가마스터는 **연 1회** full re-sync로 충분.
  - 허가/취소 → A1 실시간 API 주간 재동기화 또는 `수정일/취소일자` 증분.

### 7-6. 규모 / 저장 / 호출수 [검증/추론]
- 약가마스터 ≈ **298,183행(누적 표준코드 포함)** [검증].
- 급여 활성 품목 ≈ 2~3만 [추론], 식약처 허가 전체 ≈ 5만 품목대 [추론, 공식 통계 미확인].
- 정규화 후 수십~수백 MB → RDB 단일 인스턴스 충분.
- **호출수**: 약가마스터는 단일 CSV → API 페이지순회 불필요. 개발키 10,000콜/일 한도 이슈 없음. **결론: 파일 백본(연1 CSV + 월1 Excel) + 성분/허가 보강만 API.**

### 7-7. O4O 최소 보존 필드 [권장]
- **Raw**: 표준코드(13)·품목기준코드·제품코드(EDI)·일반명코드(주성분)·ATC·한글상품명·업체명·약품규격·단위·제형·포장형태·제품총수량·전문일반구분·품목허가일자·취소일자·상한금액·투여경로·적용일자.
- **Normalized**: PK=표준코드(stable) | drug_status(active/허가취소/급여삭제) | reimbursed(bool) | price_effective_from/to | name_history | mfds_hira_link 검증플래그 | source_snapshot_date · last_synced_at.

## 8. 비의약품 온디맨드 검색 가능성

> 비의약품은 전체저장을 기본안으로 제안하지 않음. 상품군별 검색 우선순위·저장 최소데이터·수동 폴백을 판정.

### 8-1. 의료기기 — **ON_DEMAND(바코드) 적합** [검증]
- 우선순위: **바코드(UDI-DI=GTIN) → 모델명 → 허가번호**.
- 바코드 스캔값으로 A13/A14 직접 조회 → 제품명/모델명/허가번호/제조수입업체 자동입력.
- 저장 최소: UDI-DI·품목명·모델명·허가번호·제조수입업체·등급.
- 폴백: 미등록 품목은 수기. (선택적으로 92,564행 full sync도 가능 → FULL_SYNC_PLUS_ON_DEMAND 여지.)

### 8-2. 의약외품 / 건강기능식품 / 기능성화장품 — **ON_DEMAND(제품명) 적합** [검증]
- 우선순위: **제품명 → 업체명+제품명 → 허가/보고/품목제조관리번호**. **바코드 검색 불가**.
- 선택 보조 표시필드: (의약외품) 품목명·업체명·효능; (건기식) 제품명·업체명·섭취방법·성상·품목제조관리번호; (화장품) 품목명·업소명·기준코드·SPF/PA·효능(미백/주름/자외선)·취하상태.
- 저장 최소: 공식 품목명·업체명·해당 번호(item_seq/보고일련번호/품목제조관리번호) + (화장품) 취하상태.
- 폴백: 정부 DB 미존재 시 수기. 일반(비기능성)화장품은 데이터 자체 부재 → 수기.

### 8-3. 일반식품 / 수입식품 / 위생용품 — **MANUAL_ONLY(제품명 보조검색만)** [검증]
- **바코드(GTIN) 필드 없음** → 소비자 실물 바코드로 매칭 불가. 검색축 = 제품명/업체명/품목보고번호뿐.
- 약국 즉석 등록 관점에서는 제품명 텍스트 검색 보조 외 자동등록 불가. 회수상태만 별도 API(A18/A19, A19는 바코드 검색 지원).

### 8-4. 생활용품·안전확인 제품 — **MANUAL_ONLY** [검증/추론]
- KC인증(C1)·초록누리(C2)는 식별키가 인증/신고번호이고 GTIN 매핑 없음 → 제품명 텍스트 보조검색만.

### 8-5. 전체저장이 더 적합한 비의약품 예외? [검증]
- **의료기기(92,564행)**는 규모가 작고 바코드 키가 안정적이어서 full sync도 합리적 → 유일하게 FULL_SYNC_PLUS_ON_DEMAND 후보.
- 화장품 보고품목(204,622행)도 규모상 full sync 가능하나, 바코드 부재로 자동등록 이득이 적어 ON_DEMAND 유지 권장.
- 나머지는 규모·갱신·바코드 부재로 전체저장 비권장.

## 10. O4O 적용 우선순위

### 우선순위 1 (즉시 가치, 파일 기반)
1. **처방·일반의약품 FULL_SYNC** — 약가마스터(B1, 연1 CSV) + 약제급여목록표(B2, 월1 Excel) 적재. 표준코드 PK, 식약처↔심평원 연결키 확보. 호출한도 무관.

### 우선순위 2 (온디맨드 등록 보조)
2. **의료기기 바코드 ON_DEMAND** — A13/A14, UDI-DI=바코드 직접조회. 비의약품 중 유일 자동등록.
3. **의약외품·건기식·기능성화장품 ON_DEMAND(제품명)** — A7/A8/A10 텍스트 검색 후보 제시 → 선택분만 저장.

### 우선순위 3 (보조·수기)
4. **식품·수입식품·위생용품·생활용품** — 제품명 보조검색 또는 수기. 바코드 자동등록 기대 금지.

### 설계 원칙 (요청 사업기준 반영)
- 정부 검색 실패가 상품 등록을 막지 않음(graceful degradation).
- 공급자/매장 마케팅 상품명을 정부 공식명으로 덮어쓰지 않음(공식명은 별도 필드/제안값).
- 바코드는 중복 축소용 식별 보조(POS 비목표).
- 비의약품 전체수집 비전제, 온디맨드 우선.
- 정부 이미지 확보를 핵심 선정기준으로 삼지 않음.
