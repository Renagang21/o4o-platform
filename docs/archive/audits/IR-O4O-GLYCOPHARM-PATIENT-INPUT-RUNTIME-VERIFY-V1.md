## 1. 최종 결론 (Executive Verdict)

### 사업자 피드백은 맞다.

**기술적으로** 투약/운동/증상 입력 코드는 존재하고, 저장 로직도 정상 동작한다.
**그러나 실사용 환경에서** 투약/운동/증상 입력은 사실상 불가능에 가깝다.

| 질문 | 답변 |
|------|------|
| 환자 입력 기능이 실제로 되는가? | 기술적으로 YES, 실사용 기준으로 NO |
| 사업자가 안 된다고 느끼는 이유는? | **접이식 섹션이 기본 닫힘** + **시각적 단서 부재** + **혈당 의존 구조** |
| 정확히 어느 단계가 끊기는가? | **UI 발견 단계** — 사용자가 섹션 존재를 인지하지 못함 |
| 약사 화면 표시 문제와 관계는? | **별도 문제** — 환자 입력과 무관하게 약사 화면에 metadata 미표시 |

**문제 유형: UI 숨김/가시성 문제 + UX 오해 유발 구조** (저장 실패 아님)

## 3. 문제 원인 상세

### 3.1 핵심 원인: 접이식 섹션 기본 닫힘 (Critical)

**코드 증거** — [GlucoseInputPage.tsx:87-99](services/web-glycopharm/src/pages/patient/GlucoseInputPage.tsx#L87-L99):

```typescript
const [medOpen, setMedOpen] = useState(false);   // ← 투약: 기본 닫힘
const [exOpen, setExOpen] = useState(false);      // ← 운동: 기본 닫힘
const [symOpen, setSymOpen] = useState(false);    // ← 증상: 기본 닫힘
```

**결과**: 페이지 로드 시 혈당 입력 폼만 보이고, 투약/운동/증상은 **얇은 헤더 버튼**으로만 표시된다.

**접힌 상태의 시각적 형태** — [GlucoseInputPage.tsx:301-318](services/web-glycopharm/src/pages/patient/GlucoseInputPage.tsx#L301-L318):

```
┌──────────────────────────────────────┐
│  💊 투약 기록                    ▼   │  ← 이것이 전부. 클릭 힌트 없음.
└──────────────────────────────────────┘
┌──────────────────────────────────────┐
│  🦶 운동 기록                    ▼   │  ← 동일
└──────────────────────────────────────┘
┌──────────────────────────────────────┐
│  ⚠️ 증상 기록                    ▼   │  ← 동일
└──────────────────────────────────────┘
```

- ▼ (ChevronDown) 아이콘만으로 "클릭하여 확장" 의미 전달
- "터치하여 입력" 같은 안내 텍스트 없음
- 입력 가능하다는 시각적 단서(placeholder, hint, border 강조 등) 없음

### 3.2 결정적 버그: 저장 조건이 확장 상태에 의존

**코드 증거** — [GlucoseInputPage.tsx:162-185](services/web-glycopharm/src/pages/patient/GlucoseInputPage.tsx#L162-L185):

```typescript
// 투약: 섹션이 열려 있고 AND 약품명이 있을 때만 저장
if (medOpen && medName.trim()) {
  metadata.medication = { name, dose, takenAt };
}

// 운동: 섹션이 열려 있고 AND 시간이 있을 때만 저장
if (exOpen && exDuration) {
  metadata.exercise = { type, duration, intensity };
}

// 증상: 섹션이 열려 있고 AND 증상이 선택되었을 때만 저장
if (symOpen && symptoms.length > 0) {
  metadata.symptoms = symptoms;
}
```

**`medOpen`, `exOpen`, `symOpen` 체크의 의미**:
- 섹션을 한 번도 열지 않으면 → 데이터 입력 자체가 불가능 (폼이 안 보이니까)
- 섹션을 열고 데이터를 입력해야만 → metadata에 포함됨
- 이 조건은 논리적으로 맞지만, **섹션이 기본 닫힘이라 사용자가 열지 않으므로 실질적으로 데이터가 저장되지 않음**

### 3.3 혈당 입력 필수 — 투약만 단독 저장 불가

**코드 증거** — [GlucoseInputPage.tsx:135-147](services/web-glycopharm/src/pages/patient/GlucoseInputPage.tsx#L135-L147):

```typescript
const numVal = Number(glucoseValue);
if (!glucoseValue || isNaN(numVal)) {
  setError('혈당 값을 입력하세요.');  // ← 혈당 없으면 저장 차단
  return;
}
```

**의미**: 투약만 기록하고 싶어도 반드시 혈당값을 함께 입력해야 한다. 투약/운동/증상은 혈당의 **부가 옵션**으로 설계되어 있어, 독립적 입력이 불가능하다.

### 3.4 대시보드 진입 경로 — 단일 진입점, 혈당 중심 레이블

**코드 증거** — PatientPlaceholderPage.tsx (대시보드):

```
대시보드 메뉴:
  ┌────────────────────────────────────────┐
  │  📋 데이터 입력 및 조회                │
  │     혈당·식사·운동 데이터 기록          │  ← 이 하나의 버튼이 유일한 진입점
  └────────────────────────────────────────┘
```

- 7개 메뉴 중 입력 관련 버튼은 **1개**
- "투약" 이라는 단어가 메뉴 설명에 없음 ("혈당·식사·운동" 만 표시)
- "증상" 이라는 단어도 없음
- 사용자는 이 버튼이 "혈당 입력 전용"이라고 인식할 가능성 높음

### 3.5 저장 후 재조회 — 제한적 표시

| 표시 위치 | 투약/운동/증상 표시 | 설명 |
|-----------|:------------------:|------|
| **입력 페이지 "최근 기록"** | ✅ 태그로 표시 | 약품명, 운동 시간, 증상 수 표시 |
| **데이터 분석 페이지** | ❌ 미표시 | 혈당 TIR/변동성만 분석. metadata 무시 |
| **약사 상세 페이지** | ❌ 미표시 | 타임라인 DTO에 metadata 필드 미포함 |

**입력 페이지**에서는 저장된 투약/운동/증상이 태그로 보이지만, 사용자가 주로 확인하는 **분석 페이지**에서는 전혀 보이지 않아 "저장이 안 됐다"고 느낄 수 있다.

## 6. 환자 입력 문제 vs 약사 화면 문제 — 분리 정리

### 문제 A: 환자 입력 불가 문제

| 항목 | 판정 |
|------|------|
| **원인** | UI 가시성 문제 (접이식 기본 닫힘 + 시각적 단서 부재) |
| **기술적 동작** | 섹션 확장 시 입력/저장 정상 동작 |
| **사용자 체감** | 입력 기능이 없는 것으로 인식 |
| **해결 방향** | 프론트엔드 UX 개선 (기본 펼침, 안내 텍스트, 독립 입력 등) |
| **백엔드 수정** | 불필요 |

### 문제 B: 약사 화면 미표시 문제

| 항목 | 판정 |
|------|------|
| **원인** | `PharmacistPatientDetailPage`에서 metadata 렌더링 코드 미구현 |
| **기술적 동작** | API 응답에 metadata 포함되나 화면에 미표시 |
| **사용자 체감** | 약사가 환자의 투약/운동/증상 확인 불가 |
| **해결 방향** | 프론트엔드 수정 (metadata 파싱 + 표시 섹션 추가) |
| **백엔드 수정** | 불필요 |

### 두 문제의 관계

```
문제 A (환자 입력)    ───독립───    문제 B (약사 표시)

환자가 투약을 입력하더라도 → 약사 화면에 안 보임 (B)
환자가 투약을 입력하지 못하면 → 약사 화면에도 당연히 없음 (A + B)

현재 상태: A와 B가 동시에 발생하여 "기능이 아예 없다"는 인식 강화
```

## 7. 사업자가 "안 된다"고 느끼는 구체적 시나리오 재현

```
1. 사업자가 환자 계정으로 로그인
2. 대시보드에서 "데이터 입력 및 조회" 클릭
3. 혈당 입력 폼이 크게 보임 → 혈당값 입력
4. 화면 아래쪽에 얇은 헤더 3개가 보이지만:
   - "투약 기록 ▼" — 일반 텍스트 레이블처럼 보임
   - "운동 기록 ▼" — 동일
   - "증상 기록 ▼" — 동일
5. 사업자는 이것이 "클릭하면 입력 폼이 나오는 버튼"이라고 인식하지 못함
6. 저장 버튼을 누르면 혈당만 저장됨
7. 분석 페이지에서도 혈당 데이터만 보임
8. 약사 화면에서도 혈당만 보임
9. 결론: "투약/운동/증상 입력은 안 된다"
```

**이 시나리오는 코드 버그가 아니라 UX 설계 결함이다.**
