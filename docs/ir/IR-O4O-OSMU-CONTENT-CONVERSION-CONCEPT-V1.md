# IR-O4O-OSMU-CONTENT-CONVERSION-CONCEPT-V1

> **성격: 개발 전 업무 분석 (Living Document — 단계마다 갱신).**
> 목적: "매장 콘텐츠 제작 부담 완화"를 위한 **원소스 멀티유즈(OSMU) 콘텐츠 변환** 의 개념·흐름·요구사항·진행 프로세스를 정리한다.
> 이 문서는 **본 구현의 기준점(SSOT)** 이다. 대화에서 합의된 결정·보완 요청은 **모두 기준점**이므로 단계마다 본 문서를 갱신한다.
> 일부는 확정(✅ 합의), 일부는 미결(공급자/시장 의견 청취 대상)로 표기한다.
> 작성일: 2026-06-23 · 최근 갱신: 2026-06-23 · 출처: 사업 구상 대화 정리

## 0. 한 줄 요약

> **매장 사업자가 콘텐츠(동영상·POP·QR·블로그 등)를 만드는 일이 너무 무겁다.**
> 기존 AI(Gemini) 자동 생성은 통제가 안 돼 실패했다.
> 대안: **하나의 잘 정리된 소스 → 각 사용처로 변환(OSMU)** 구조를 만들되,
> **원 소스(골격) 제작은 제품을 가장 잘 아는 공급자에게 맡기고, o4o는 표준·검수만 소유한다.**

---

## 1. 문제 정의

- 매장 실행 자산은 이미 6종으로 정의돼 있다: **상품상세 / POP / QR / 블로그 / 사이니지 / 고객안내문** (Store Menu Canonical Tree V1).
- 각 사용처에는 **템플릿(그릇)이 이미 있다.**
- 그러나 매장 사업자가 **기본 콘텐츠를 받아 각 사용처용 자산으로 바꾸는 일**은 여전히 만만치 않다.
- 기존 **Gemini API 자동 생성**은 실사용에서 문제가 많았다.
  - 진짜 원인: "AI라서"가 아니라 **비결정적(non-deterministic)** 이라서.
  - 매번 결과가 다르고, 레이아웃·브랜드 통제가 안 되고, "이렇게 고쳐줘"가 어렵다.
- 사업자가 진짜 원하는 것은 **창작이 아니라 "내 정보만 넣으면 항상 똑같이 깔끔하게 나오는 것"** 이다.

## 2. 핵심 통찰 (대화에서 합의된 것)

### 2.1 빠진 조각 = 변환 계층

```
[소스 콘텐츠]  →  ??? (없음)  →  [QR 템플릿]
   (이미지/문서)                  [POP 템플릿]
                                 [블로그 템플릿]
                                 [동영상]
```

각 사용처에 템플릿(그릇)은 있으나, **소스를 각 그릇에 맞게 덜어 담는 "변환 계층"** 이 없다. 이것이 OSMU의 핵심이자 첫 단계다.

### 2.2 변환 = "모양 바꾸기"가 아니라 "분량 덜어내기"

각 사용처는 필요한 분량·깊이가 완전히 다르다.

| 사용처 | 소스에서 가져가는 것 | 성격 |
|---|---|---|
| QR | 링크 1개 | 포인터 |
| POP | 핵심 메시지 1줄 + 가격 + 이미지 | 극도 압축 |
| 블로그 | 전체 서사 | 풀어 쓰기 |
| 동영상 | 슬라이드 순서 + 타이밍 | 시간축 재배열 |
| 상품상세 | 스펙 구조화 | 항목별 정리 |

→ **하나의 풍부한 소스에서 각 채널이 "필요한 만큼만 잘라서 제 모양으로" 가져간다.**

### 2.3 PPT/문서가 소스로 적합한 이유

- 사람은 PPT/문서로 정리할 때 생각이 가장 잘 정리된다.
- 분량이 많은 콘텐츠는 단일 문서로 전달이 어렵다 → **PPT → 동영상** 으로 옮기면 핵심 정리가 잘 된다.
- PPT는 이미 **슬라이드 단위로 구조화(제목·핵심·이미지)** 되어 있어, 변환 계층이 "어디서 무엇을 뽑을지" 알기 쉽다. (줄글보다 다루기 좋은 소스)

### 2.4 AI의 올바른 자리 = "창작"이 아니라 "변환/추출"

- 기존 Gemini 실패: AI에게 **무에서 창작**을 시킴 → 통제 불가.
- 올바른 사용: **이미 있는 소스 → 정해진 그릇(스키마 슬롯)** 으로 압축·재배열·추출만.
- 입력도 출력도 경계가 분명 → 결과 예측 가능, AI는 보조 역할로 묶임.

### 2.5 지배 원칙

> **"엔진은 빌려도 그릇은 내가 가진다."**
> 그리고 그 확장: **"작업(원 소스 제작)은 공급자에게 맡겨도 그릇(골격의 모양)은 o4o가 가진다."**

추출 AI도, 모델 구축도 o4o가 직접 만들지 않는다. o4o는 **표준과 게이트만** 소유한다.

## 3. 합의된 방향 (현재 가설)

### 3.1 진짜 소스 = 이미지로 된 상품 상세 설명서

- 현재 가장 많은 제품 콘텐츠 = **이미지 상품 상세 설명서**. 모든 사업자가 보유.
- 이를 각 사용처로 변환하면서 **매장 경영자의 생각이 들어가는 것**이 시장성이 가장 좋다.
- 가장 어려운 건 "원 소스 만들기" 인데, **상품 설명서 + 공급자 추가 자료**면 필요한 콘텐츠 내용은 사실상 충분하다.

### 3.2 제안 흐름 (사장 구상)

```
상품 상세 설명서(이미지) + 추가 자료(공급자)
   → 기본 콘텐츠(골격) 만들기
   → 매장 경영자/운영자 의견 추가
   → 정리
   → 각 사용처에 맞는 방식으로 변환
   → 사용처에서 템플릿 선택 후 변환
   → 간단한 수정
```

### 3.3 이 흐름은 o4o의 3자 Canonical Flow와 일치한다

```
공급자: 이미지+자료 → 골격 제출        (원 소스 = 제품을 가장 잘 아는 사람이)
   ↓
o4o:   검수 → 의견 회신 → 재정리 요청   (← 이미 있는 "검수·승인" 워크플로우 재사용)
   ↓
공급자: 보완 제출 → 승인
   ↓
매장:  골격 받아 의견 추가 → 각 사용처 변환
```

특히 **"점검 → 의견 회신 → 재정리"** 는 o4o에 **이미 구현된 검수·승인 루프**(approve / reject / revision_requested)와 동일하다. **신규 구축이 아니라 재사용**.

### 3.4 왜 공급자에게 골격 제작을 맡기는가

- **지식**: 제품을 어떻게 팔지는 공급자가 가장 잘 안다.
- **동기**: 공급자는 자기 제품이 팔리길 원함 → 시키지 않아도 잘 만들 이유가 있음.
- **확장성**: o4o가 모든 골격을 직접 만들면 병목. 공급자가 만들면 무한 확장.

## 4. o4o가 소유·구축해야 하는 것 (딱 3가지)

| # | 항목 | 설명 |
|---|---|---|
| 1 | **표준 골격 정의** | 공급자가 채워야 할 칸 = 스키마. **사실/표현 분리 포함.** |
| 2 | **검수 기준 (체크리스트)** | 무엇이 통과/반려인가: 필수칸 채움 / 사실분리 / 이미지품질 / 과장표현 없음 |
| 3 | **피드백 루프** | 이미 있는 검수·승인(approve/reject/revision) 재사용 |

→ 추출 AI·자체 모델은 만들지 않는다. **표준과 게이트만** 가진다 → 가볍다.

## 5. 절대 양보 불가 사항

### 5.1 "가이드/프롬프트 예제"는 반드시 "규격서(스키마)"여야 한다

- 느슨한 권유 → 공급자마다 제각각 모양 제출 → OSMU 변환 불가 (Gemini 실패 재발).
- 엄격한 양식 → 공급자가 **정해진 칸을 채움** → 항상 같은 그릇 → 변환 자동화 가능.
- 즉 공급자에게 줄 "프롬프트 예제"는 사실상 **"이 양식대로 채워서 내라"는 규격서**다.

### 5.2 사실(Fact)과 표현(Expression)을 처음부터 칸으로 분리

- **사실 칸**: 효능·성분·스펙 (불변, 규제 대상)
- **표현 칸**: 판매 포인트·매장 의견 (가변)
- 이유: 화장품·글라이코팜 등 **표현 규제 도메인**이 있어, 매장 의견이 효능을 과장하면 **과대광고 위반**.
- 나중에 고치기 매우 어려운 구조 → **첫 설계에 반드시 포함**.

### 5.3 매장 의견도 "구조적으로" 받는다

- 줄글로 받으면 변환 불가.
- "이 제품의 강조점 / 우리 매장 타겟 / 가격·이벤트" 처럼 **칸으로** 받아야 각 채널로 흘러간다.

## 6. AI 사용 방식 옵션 (참고 — 변환 계층 구현 시)

| 방식 | 누가 AI 호출 | UX | 구조 유지 | 평가 |
|---|---|---|---|---|
| A. 프롬프트만 주고 사용자가 외부 서비스 이용 (복붙) | 사용자 | 나쁨 | ❌ 깨짐 | **수요 검증용 0차 실험으로만** |
| B. o4o가 외부 API 호출 + o4o 프롬프트/스키마 강제(Structured Output) | o4o | 좋음 | ✅ 유지 | **권장** |
| C. o4o 자체 모델 구축 | o4o | 좋음 | ✅ | 과함·불필요 |

- 단기 수요 검증만이면 A → 수요 확인 후 B로 전환.
- AI의 또 다른 자리: **검수 보조** (제출 골격을 표준과 대조해 "필수칸 누락 / 효능 과장 의심" 검수 의견 초안 생성). "창작"이 아니라 "규격 대조"라 통제 가능 → 운영자 검수 부담 경감.

## 7. 강제력 (게이트 연결)

- 공급자가 양식을 따를 이유 = **"골격 제출 = 제품 등록(유통 참여)의 전제조건"** 으로 묶는다.
- o4o엔 이미 공급자 온보딩·제품 등록 gate 존재 (온보딩 3단계 완료, 다음이 제품 등록 gate).
- 거기에 "골격 제출" 한 칸을 얹는 형태.

## 8. 흐름 보정 결정 (✅ 합의 — 2026-06-23)

매장 콘텐츠 제작 흐름을 다음 3가지로 보정한다.

### 8.1 원본(Source) vs 출력물(Output) 분리

| | 원본(Source) | 출력물(Output) |
|---|---|---|
| 정의 | 편집 가능한 제작물 | 디바이스에 바로 쓰는 렌더링 결과 |
| 예 | pptx, 골격, 문서 초안 | MP4, 반응형 HTML, POP 이미지(PNG), QR |
| 성격 | 수정·재사용 | 배포·소비 |

→ 흐름: **① 원본 제작 → ② 디바이스/포맷별 출력물 렌더링 → ③ o4o 저장·배포**
"o4o 지원 언어로 변환" = ②의 출력 렌더링 계층. 원본은 두고 출력만 재생성 가능해야 함.

### 8.2 동영상은 "반응형"이 아니라 "비율 선택"

- **문서(POP·블로그·상품상세)**: HTML 반응형 → 디바이스 자동 대응.
- **동영상**: 고정 비율 파일 → 목표 비율을 **제작 시작 시** 정해 각각 렌더링.

**디바이스 → 비율 표준 세트 (✅ 합의 — 2026-06-23):**

| 비율 | 디바이스 | 변환 모드 |
|---|---|---|
| **16:9 가로** | PC·TV·가로 사이니지·유튜브 | base(원본 비율) |
| **9:16 세로** | 고객 폰(QR)·세로 사이니지·세로 키오스크 | STACK(상단 텍스트+하단 이미지 세로 스택) |
| **16:10 가로** | 태블릿(안드로이드, 가로) — 매장 응대 다국어 표시 | FIT(16:9 레이아웃 균일 축소·중앙, 좌우배치 보존) |
| **1:1 정사각** | SNS 피드(인스타·페북) — **파트너 마케팅용**(향후) | STACK |
| (3:4·4:5 세로) | iPad 세로·SNS 세로 | STACK (필요시) |

- **태블릿은 가로(16:10)** 기준. SNS 피드(1:1)는 아직 아니나 **파트너가 사용 예정**이라 함께 생성.
- 배경은 §10.3 **크롭 우선**(목표 비율 기준 중앙 크롭, 별도 이미지 제작 없음).
- **✅ 결정: o4o 동영상 = "pptx + 애니메이션" 으로 한정.** 자유 영상편집 미지원 → 결정적(deterministic) 품질 보장.

### 8.3 저장 소유권 = 매장(조직) 경계

- "매장 경영자 아이디로 저장"의 정확한 의미: **소유 = 매장(`storeId`/`organizationId`), 행위자 = 매장 경영자.**
- 근거: Boundary Policy(Store Ops 경계 = `organizationId`), F11(개인 user 속성으로 소유·권한 묶기 금지).
- 이유: 경영자 교체·직원 제작 시에도 매장 자산은 보존되어야 함.
- 저장 2계층: **공용 원본**(공급자/Operator 발) — 공유 / **매장 제작 결과물** — 매장 소유(기존 Store Production Material 모델).

## 9. Store 실행계층 정렬 결정 (✅ 합의 — 2026-06-23)

### 9.1 핵심 결정

> **저작(콘텐츠 제작)은 운영자(Operator)가 한다. 매장 경영자는 가져다 그대로 쓰거나 살짝 수정한다.**

- 이는 단순 기능 축소가 아니라 **Store를 원래 자리(실행 계층)로 되돌리는 정렬**이다.
- o4o 원래 모델: **HUB = 공급 / Store = 실행**, 흐름 = **"선택 → 복사 → 사용처 → 실행"** (`asset-copy-core` 존재).
- 현재 내 매장의 무거운 제작 기능은 **Store가 제작 계층으로 번진 Drift** 가능성이 큼.

### 9.2 내 매장 기능 재정의

| 구분 | 처리 |
|---|---|
| **대폭 축소** | 빈 캔버스 창작, 복잡한 에디터, 무에서 제작 |
| **남김/강화** | 운영자 자산 **갤러리** / **가져오기(복사)** / **빈칸 채우기 경량 편집(가격·강조점·타겟·매장 특색)** / **사용처 배치(사이니지·POP·QR)** / **저장·관리** |

### 9.3 "막막함" 해소 원칙

> 첫 화면 = **"만들기"가 아니라 "고르기"** — "우리 매장에 쓸 수 있는 자료가 준비돼 있습니다 → 고르세요"

### 9.4 주의 (축소 ≠ 제거)

1. **매장의 "생각"(차별화)은 보존** — 경량 편집(빈칸 채우기)은 반드시 남긴다. 사라지면 "남의 콘텐츠 그대로 쓰기"가 되어 차별화 소멸.
2. **2단계 제공** — 기본(그대로 사용, 초보 막막함 0) / 고급(수정, 파워유저 비차단).
3. **기존 매장 자료 보호** — 현행 제작물 마이그레이션 고려.

### 10.3 디바이스별 제작 & 배경 이미지 정책 (✅ 합의 — 2026-06-23, **갱신**)

- 디바이스(비율)마다 동영상 별도 제작: 16:9 가로 / 9:16 세로 / 4:3 등. (동영상은 반응형 아님 — §8.2)
- **전경(텍스트·제품 이미지)**: 위치·크기 **리플로우(프로그램 재배치)** — 비율마다 자동 재배치.
- **배경(페이지 전체 이미지)**: 비율 변경 시 늘리면 왜곡 / 자르면 손실.

**결정(정정) — 배경은 "크롭 우선"이 상당 기간 기본:**
- 초기엔 "비율별 별도 등록"을 검토했으나, 현실적으로 **상당 기간은 배경을 디바이스 크기에 맞게 크롭(중앙/안전영역 기준 잘라내기)** 하여 쓴다. 매 비율마다 배경을 새로 만들지 않는다.
- **공급자·운영자 디자인 가이드 (필수 전달):** "배경 이미지는 **디바이스 크기에 따라 잘릴 수 있으니 그것을 감안해 디자인하라** — 핵심 시각 요소(로고·제품·핵심 카피)는 **화면 중앙의 안전 영역(safe area) 안**에 두고, 가장자리에는 잘려도 무방한 여백/패턴만 둔다." (= 공급자 골격 규격서 §4·§5 에 배경 안전영역 항목 추가)
- **Fallback (예외):** 크롭으로 핵심이 손실되는 구도(가로 전용 등)에 한해 **해당 비율 배경을 별도 제작·등록**. 기본 아님.
- 원칙: **전경 = 자동 재배치 / 배경 = 크롭 우선(안전영역 디자인) · 별도 등록은 예외.**

|---|---|---|
| R1 | 고정 슬롯 | 텍스트는 박스가 아니라 고정 영역(위치+폭+높이). `spAutoFit` 폐기 → `normAutofit`. |

- **EN(최장)** — 내용과다 슬라이드(7·8·10·11)에서 텍스트 축소(0.76~0.80) 발생. 길이 증가분을 텍스트 축소 + 이미지 밀기/축소로 분담.
- **ZH(최단, 0.78x)** — slide12 외 **전 슬라이드 무축소(1.0)**. 짧아진 언어는 리플로우 거의 불요 → 여백 증가. "겹침 없음 + 읽을 만함" 자동 충족.
- **JA(중간, 1.01x)** — slide11(0.96, 미세)·slide12만 축소. KO 와 길이 비슷해 거동도 최소.
- **slide12 는 세 언어 모두 0.84** — 4개 텍스트 블록(제품명/효능/문구/연락처)이 구조적으로 빽빽 → **언어 무관 구조적 제약**. 솔버가 언어구동 축소(7·8·10·11)와 구조적 축소(12)를 분리해 처리함을 확인.
- 구현: CJK 는 전각이라 `merge.py` 글자폭 `CW=1.0`(라틴 0.55), 텍스트 간격은 언어 무관하게 `spcBef` 가 정확 처리(추정 불요, R11). 파일: `자일리톨그린_동영상_{EN,ZH,JA}_16x9.pptx`.
- 언어 선택: `OSMU_LANG=en|zh|ja` 환경변수(`trans.py`·`merge.py`). EN 기본값 보존.

**간격 표준화 보정 + CJK orphan (2026-06-23, EN/ZH/JA 재적용):** 사용자 검수에서 "제목(큰 글자)–부제(작은 글자) 시각 간격이 슬라이드마다 다르다 / CJK 마지막 1글자 줄넘김" 확인 → 근본 원인·해법 정정.
- **원인:** PowerPoint 는 `spcBef` 위에 **폰트크기 비례 leading** 을 더해 렌더 → **상수 spcBef(10pt) 로는 시각 간격이 절대 균일해지지 않음**(제목44 + 부제24/28/44 … 조합마다 달라짐). R11 의 "동일 spcBef" 가정의 허점.
- **해법(간격):** 경계별 `spcBef = GAP_VISUAL − LEAD·(prev_fs + next_fs)` 로 **폰트크기 역보정** → 시각 간격을 상수(GAP_VISUAL≈22pt)로 고정. 큰 폰트일수록 spcBef 작게. 실측: 44→24=8pt, 36→44=6pt, 44→28=7pt, 24↔28=11pt — **시각 간격 균일**. (params GAP_VISUAL/LEAD 튜너블)
- **해법(orphan, 정정):** 초기엔 폰트 축소(deorphan)로 대응했으나 — 사용자 지적("칸이 좁아 넘침; 문맥 보고 줄바꿈해야") 이 정답. **근본 원인:** 원본 PPTX 는 이미 `<a:br>` 로 디자이너 의미단위 줄바꿈을 해뒀는데 단순 치환이 이를 버려 PowerPoint 자동 줄바꿈(아무 데서나 끊김 → CJK 1글자 orphan)에 맡김. **해법:** 번역을 **의미 단위 줄 목록**(AI 가 언어·폭에 맞게 결정)으로 주고 `<a:br>` 명시 줄바꿈 — `trans.py` 문단값 `str`(한 줄)|`list`(여러 줄, `<a:br>` 연결, rPr 보존). 예: 美味持久，\|唾液也自然分泌(쉼표), 采用乐天Wellfood的\|口香糖技术制成(的 뒤). **폰트 안 줄이고 orphan 제거 + 자연스러운 끊김**(slide1 36pt·slide2 44pt 그대로). font-shrink(`deorphan`)는 세그먼트가 그래도 넘칠 때의 **폴백**으로 강등(`<a:br>` 단위 폭 평가).
  - **PoC 결론(역할 분담):** 줄바꿈은 **언어 이해가 필요 → AI(번역 단계)의 일**. 메커니컬 스크립트(정규식)는 의미를 못 읽어 자동 줄바꿈/폰트 축소만 가능. 운영 툴은 **AI 가 번역 + 언어별 의미단위 줄바꿈을 함께 산출**, 결정적 툴은 배치/애니만 담당.
- ⚠️ 간격 보정은 추정 기반(leading/폭) — 실 렌더 미세차 가능, 사용자 시각 검수 + param 튜닝 전제. `merge.py` `comp_spcbef`(간격)/`deorphan`(orphan 폴백) · `trans.py`(의미단위 세그먼트).

## 11. 작업 방법론 (✅ 합의 — 2026-06-23)
