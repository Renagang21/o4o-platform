# WO-SERVICE-PARTNER-DASHBOARD-UI-SKELETON-V1

**(서비스별 파트너 대시보드 UI Skeleton 정의)**

---

## 1. Work Order 목적 (Purpose)

본 Work Order의 목적은 다음과 같다.

* **WO-SERVICE-PARTNER-DASHBOARD-BASELINE-V1**에서 확정된
  파트너 책임·금지 사항을
  실제 **화면 구조(UI Skeleton)** 로 고정한다.
* 서비스별 구현 편차가 발생하더라도
  "틀린 구현"과 "허용된 변형"을 즉시 구분할 수 있는
  **구조 기준선**을 제공한다.
* 이후 프론트엔드 작업 요청 시
  **화면 구성 논의 없이 바로 구현 가능**한 상태를 만든다.

본 문서는
**디자인 가이드도 아니고, 컴포넌트 명세도 아니다.**
👉 *"어떤 화면이 존재하며, 무엇이 있고, 무엇이 없어야 하는가"만 정의한다.*

---

## 2. 적용 범위 (Scope)

### 적용 대상

* 모든 **서비스 내부 파트너 대시보드**

  * GlycoPharm
  * K-Cosmetics
  * GlucoseView
  * 향후 추가 서비스

### 적용 제외

* Neture (파트너 안내 허브)
* 관리자(Admin) 화면
* 공급자(Supplier) 대시보드
* 판매자(Seller) 대시보드

---

## 3. 상위 기준 문서 (Normative References)

본 Work Order는 아래 문서를 **절대 기준**으로 한다.

* **WO-NETURE-PARTNER-DASHBOARD-RESPONSIBILITY-V1**
* **WO-SERVICE-PARTNER-DASHBOARD-BASELINE-V1**

본 문서의 모든 UI 구성은
위 문서의 **Hard Rules를 침해할 수 없다.**

---

## 4. 파트너 대시보드 전체 구조 (Top-level Layout)

### 공통 최상위 구조

```
Partner Dashboard
 ├─ Overview (요약)
 ├─ My Promotion Targets (홍보 대상)
 ├─ Content (콘텐츠)
 ├─ Events (이벤트 조건)
 └─ Status (상태)
```

다음 항목은 **구조적으로 존재하지 않는다**:

* 승인(Approval)
* 성과(Analytics)
* 운영 설정(Operation)
* 사용자/권한 관리

---

## 5. 섹션별 UI Skeleton 정의

### 5.1 Overview (요약)

**목적**
파트너가 "지금 무엇을 하고 있는지"를
**판단 없이 인지**할 수 있도록 한다.

**포함 요소**

* 현재 서비스 명
* 활성 콘텐츠 수
* 진행 중 이벤트 수
* 전체 상태 요약 (활성 / 비활성)

**금지 요소**

* 성과 수치
* 매출 기여
* 클릭/전환 지표
* 승인 상태 표시

👉 *정보 요약 화면이지, 판단 화면이 아니다.*

---

### 5.2 My Promotion Targets (홍보 대상)

**목적**
파트너가 **홍보 대상이 무엇인지 명확히 인식**하도록 한다.

**포함 요소**

* 연결된 매장 / 판매자 목록
* 대상 서비스 영역 설명
* "이 대상은 서비스에 의해 지정됨"이라는 명시 문구

**금지 요소**

* 대상 추가/삭제 UI
* 대상 승인 요청
* 대상 우선순위 조정

👉 *선택 화면 (X) / 인지 화면 (O)*

---

### 5.3 Content (콘텐츠)

**목적**
파트너가 **홍보용 콘텐츠를 작성·관리**할 수 있게 한다.

**포함 요소**

* 콘텐츠 목록
* 콘텐츠 생성 / 수정 / 비활성
* 콘텐츠 타입

  * 텍스트
  * 이미지
  * 링크

**명시적 제한**

* 상품 정보 참조는 "읽기 전용"
* 가격 / 재고 / 옵션 접근 (X)

**UI 주의 문구 (필수)**

> "콘텐츠의 실제 노출 방식과 위치는
> 서비스 정책에 따라 결정됩니다."

---

### 5.4 Events (이벤트 조건)

**목적**
파트너가 **이벤트의 조건만 설정**하도록 한다.

**포함 요소**

* 이벤트 목록
* 기간 설정
* 지역 설정
* 대상 범위 설정

**명시적 제외**

* 이벤트 콘텐츠 설명
* 이벤트 적용 결과 미리보기
* 이벤트 성과 예측

👉 *조건 입력 UI / 실행 판단 UI (X)*

---

### 5.5 Status (상태)

**목적**
파트너가 **현재 상태를 "확인만"** 할 수 있도록 한다.

**포함 요소**

* 콘텐츠 상태: 활성 / 비활성
* 이벤트 상태: 진행 중 / 종료
* 대상 적용 여부 표시

**금지 요소**

* 성과 그래프
* 비교 지표
* 추천 / 최적화 안내

---

## 6. 전역 UI 금지 규칙 (Hard UI Rules)

다음 요소는 **모든 서비스 파트너 대시보드에서 금지**한다.

* "승인 요청", "검토 중" 등 승인 암시 표현
* "성과", "기여도", "효과" 용어
* 관리자 화면과 동일한 레이아웃/컴포넌트
* 운영자 시점의 CTA 버튼

위반 시:

> **UI Skeleton 위반으로 즉시 수정 대상**

---

## 7. 서비스별 허용 변형 (Allowed Variations)

서비스는 다음만 자유롭게 변경 가능하다.

* 섹션 명칭
* UI 배치 순서
* 콘텐츠 카드 표현 방식

변경 불가:

* 섹션 존재 여부
* 책임 경계
* 금지 항목

---

## 8. Work Order 종료 조건 (DoD)

다음이 충족되면 본 Work Order는 완료로 간주한다.

* 모든 서비스 파트너 대시보드가
  본 Skeleton 구조를 기준으로 설계 가능
* 프론트엔드 작업 요청 시
  "이 화면이 왜 필요한가" 논의가 사라짐
* Hard Rule 위반 여부를
  **UI만 보고 판별 가능**해짐

---

## 9. 최종 기준 문장

> **파트너 대시보드는
> "무엇을 할 수 있는지"가 아니라
> "어디까지 할 수 있는지"를 보여주는 화면이다.**
