# O4O Shared Space Frame Principle v1

## 1. 배경

Forum, Content, Signage, LMS, Market Trial 등은 각각 독립된 기능과 책임을 가진 앱 성격의 파트이며, 이들은 core + extension 구조로 설계·확장한다.

반면 공동 공간은 하나의 기능 앱이 아니라, 여러 기능을 연결하고 요약하고 안내하는 공간이다.

따라서 공동 공간을 app core + extension 구조로 직접 정의하지 않고, 별도의 구조 원칙으로 다룬다.

---

## 2. 핵심 원칙

공동 공간은 app core가 아니라 space frame으로 본다.

공동 공간은 다음 역할을 가진다:

* 여러 기능을 한 화면 안에서 조합한다
* 현재 서비스 상태를 요약한다
* 사용자에게 다음 행동을 안내한다
* 기능 간 연결을 자연스럽게 만든다

공동 공간은 실행 기능의 중심이 아니라, 탐색·연결·요약·진입을 담당하는 상위 구조이다.

---

## 3. 기능 앱과 공동 공간의 구분

### 3.1 기능 앱 (core + extension 대상)

다음 영역은 기능 중심 구조를 가지므로 core + extension으로 설계한다:

* Forum
* Content / Library
* Signage
* LMS
* Store 실행 기능
* 기타 도메인 기반 기능 앱

이들은 자체적으로:

* 데이터 구조
* 목록/상세/생성/편집 흐름
* 권한 및 상태
* 도메인 규칙

을 가진다.

---

### 3.2 공동 공간 (space frame 대상)

공동 공간은 기능을 직접 수행하지 않는다.

대신 다음을 담당한다:

* 공지 / 뉴스 / 외부 정보 노출
* 최근 활동 요약
* 기능 앱 진입 카드 제공
* 추천 콘텐츠 노출
* 외부 서비스 연결
* CTA 구성
* 개인화 또는 운영자 노출 블록

즉, 공동 공간은 기능 실행이 아니라 기능 연결과 안내를 담당한다.

---

## 4. 구조 원칙

공동 공간은 core + extension이 아니라 다음 구조로 설계한다:

* Space Frame
* Section Modules
* Service Composition

예시 구조:

```
Shared Space Frame
├─ Hero / Summary
├─ News Module
├─ Activity Module
├─ App Entry Module
├─ Recommendation Module
├─ Utility Module
```

각 서비스는 이 프레임 위에서 모듈을 조합한다.

---

## 5. 표준화 대상

공동 공간에서 표준화할 대상은 기능이 아니라 공간 문법이다.

표준화 대상:

* 섹션 배치 구조
* Hero 영역 위치
* 뉴스/공지 영역 위치
* 최근 활동 영역 위치
* 기능 진입 영역 위치
* SectionHeader 규칙
* 전체보기 위치
* 카드 간격 및 정렬
* empty / loading / error 상태 표현
* 반응형 기준

---

## 6. 서비스별 차별성

공동 공간은 프레임을 공유하되 서비스별 차이를 허용한다.

허용 범위:

* 섹션 순서
* 강조 영역
* CTA 문구
* 용어
* 카드 타입
* 노출 밀도
* 외부 연결 요소

---

## 7. 기대 효과

* 공동 공간의 비대화 방지
* 기능 앱의 독립성 유지
* 서비스별 차별성 확보
* UI-UX 일관성 확보
* 신규 서비스 확장 속도 향상
* AI 기능 결합 용이

---

## 8. 문서 운영 규칙

이 문서는 플랫폼 공통 설계 기준이다.

다음 경우에만 수정 가능하다:

* Core/Extension 구조 자체 변경 시
* Space Frame 구조가 실제 운영에서 문제를 일으킨 경우
* 새로운 공통 패턴 등장으로 기존 구조로 수용 불가능한 경우

그 외 변경은 금지한다.

---

## 9. 결론

공동 공간은 기능 앱이 아니다.

공동 공간은 여러 기능을 연결하고 안내하는 space frame이며,
기능 앱은 core + extension 구조를 유지하고,
공동 공간은 이를 조합하는 구조로 설계한다.
